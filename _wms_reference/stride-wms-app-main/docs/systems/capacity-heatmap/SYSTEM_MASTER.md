## 1) DR-1 FUNCTIONAL SCOPE

### Purpose
Capacity & Heat Map provides a **visual warehouse-map overlay** that colors map nodes by zone utilization and supports zone drill-down to location utilization details.

- UI map overlay and drill-down are provided by `WarehouseHeatMap` with zone and location rows.  
  **Evidence (Confirmed):** `src/pages/WarehouseHeatMap.tsx` (`WarehouseHeatMap`, imports and usage of `useWarehouseMapZoneCapacity` + `useWarehouseMapLocationCapacity`, plus SVG render and zone detail panel).
- Zone-level and location-level utilization values are sourced by RPC-backed hooks.  
  **Evidence (Confirmed):** `src/hooks/useWarehouseMapZoneCapacity.ts`; `src/hooks/useWarehouseMapLocationCapacity.ts`; `src/lib/capacity/capacityModule.ts` (`fetchWarehouseMapZoneCapacity`, `fetchWarehouseZoneLocationCapacity`).

### Explicit non-goals
- Geometry authoring internals are outside this system slice (owned by map builder/editor surface).
- External map providers, GPS, and reporting engine are out of scope by prompt.
- Billing rate calculations are out of scope except storage-eligibility interaction risk notes.

### Cross-system touchpoints
- **Inventory:** Uses inventory volume primitives (`inventory_units`, `containers`) in heat-map RPC aggregation.
  **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql` (`rpc_get_warehouse_map_zone_capacity` CTEs for `inventory_units` and `containers`); `supabase/migrations/20260219150000_hmv_p1_zone_location_capacity_rpc.sql`.
- **Locations:** Uses `locations.zone_id`, `locations.capacity_cu_ft/capacity_cuft`, and active-location filters.
  **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `src/integrations/supabase/types.ts` (`locations` shape).
- **Warehouse Map:** Depends on `warehouse_maps`, `warehouse_map_nodes`, default map behavior, and route wiring.
  **Evidence (Confirmed):** `src/hooks/useWarehouseMaps.ts`; `src/App.tsx`; `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`.
- **Billing:** No direct heatmap-billing RPC coupling found; billing storage generation is separate (`generate_storage_for_date`).
  **Evidence (Confirmed):** `docs/systems/billing/SYSTEM_MASTER.md` (storage RPC references).  
  **Status:** Potential semantic mismatch risk only (see DR-7).
- **Receiving:** Receiving updates item locations; capacity-related recalculation for heatmap is DB/RPC-side and not event-pushed to heatmap UI.
  **Evidence (Confirmed):** `src/hooks/useReceivingSession.ts` (`rpc_assign_receiving_location_for_shipment`, item `current_location_id` updates); `src/pages/WarehouseHeatMap.tsx` (manual refresh action).

## 2) DR-2 ARCHITECTURAL WIRING

### Trace 1 — Warehouse map load (heatmap screen)
`WarehouseHeatMap page → useWarehouseMaps hook → supabase.from('warehouse_maps') + rpc_set_default_warehouse_map → default map selected in page state → SVG rendered with map dimensions`

- **Evidence (Confirmed):** `src/pages/WarehouseHeatMap.tsx`; `src/hooks/useWarehouseMaps.ts`; `supabase/migrations/20260219130000_rpc_set_default_warehouse_map.sql`.

### Trace 2 — Capacity aggregation load (zone overlay)
`WarehouseHeatMap → useWarehouseMapZoneCapacity → fetchWarehouseMapZoneCapacity → Supabase RPC rpc_get_warehouse_map_zone_capacity(p_map_id) → SQL CTE aggregation by zone/map node → rect fill + legend render`

- **Evidence (Confirmed):** `src/pages/WarehouseHeatMap.tsx`; `src/hooks/useWarehouseMapZoneCapacity.ts`; `src/lib/capacity/capacityModule.ts`; `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`.

### Trace 3 — Heat intensity calculation
`WarehouseHeatMap zone rows → getHeatFill(utilization_pct) → fixed bucket color mapping → SVG rect fill`

- **Evidence (Confirmed):** `src/pages/WarehouseHeatMap.tsx` (uses `getHeatFill`); `src/lib/heatMapUtils.ts` (`>100`, `>=80`, `>=50`, else green, null gray).
- **Logic location:** Client-side visual mapping only; utilization_pct itself is computed server-side in RPC.

### Trace 4 — Location selection interaction
`Zone rect onClick → selectedZoneId state update → useWarehouseMapLocationCapacity(mapId, zoneId) → rpc_get_warehouse_zone_location_capacity → right-panel location list sorted/filtered`

- **Evidence (Confirmed):** `src/pages/WarehouseHeatMap.tsx` (`onClick` on `<g>`, `selectedZoneId`, `showOnlyAvailable`); `src/hooks/useWarehouseMapLocationCapacity.ts`; `src/lib/capacity/capacityModule.ts`; `supabase/migrations/20260219150000_hmv_p1_zone_location_capacity_rpc.sql`.

## 3) DR-3 DATA MODEL (CAPACITY/HEATMAP SLICE)

### Core tables and fields

1. **`locations`**
   - Capacity-related fields: `capacity`, `capacity_sq_ft`, `capacity_cu_ft`, `capacity_cuft`, `length_in`, `width_in`, `usable_height_in`.
   - Grouping fields: `zone_id`, `group_code`.
   - Tenant scoping path: via `warehouse_id` (no direct `tenant_id` field in typed row).
   - **Evidence (Confirmed):** `src/integrations/supabase/types.ts` (`locations.Row`); `supabase/migrations/20260212160000_location_capacity_suggestion_engine.sql`; `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260212170000_location_capacity_patch_grouping_flags.sql`.

2. **`warehouse_zones`**
   - Fields: `tenant_id`, `warehouse_id`, `zone_code`, `description`, `sort_order`.
   - **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`.

3. **`warehouse_maps`**
   - Fields: `tenant_id`, `warehouse_id`, `name`, `width`, `height`, `grid_size`, `is_default`.
   - **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`.

4. **`warehouse_map_nodes`**
   - Fields: `warehouse_map_id`, `zone_id`, geometry (`x`,`y`,`width`,`height`), `label`, `group_label`.
   - Zone alias/group labeling are persisted at node level (`label`, `group_label`), not location rows.
   - **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260219160000_hmv_p1_map_node_group_labels.sql`; `src/integrations/supabase/types.ts` (`warehouse_map_nodes.Row`).

5. **`location_capacity_cache`**
   - Fields: `location_id` (PK/FK), `used_cuft`, `available_cuft`, `utilization_pct`, `updated_at`.
   - **Evidence (Confirmed):** `supabase/migrations/20260212160000_location_capacity_suggestion_engine.sql`; `src/integrations/supabase/types.ts`.

6. **Item linkage and volume source**
   - `items.location_id` and `items.current_location_id` appear in client capacity helper logic.
   - Heatmap RPCs primarily aggregate from `inventory_units` and `containers` (not directly `items`).
   - **Evidence (Confirmed):** `src/lib/capacity/capacityModule.ts`; `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260219150000_hmv_p1_zone_location_capacity_rpc.sql`.

### Nullability/defaults/FKs/checks
- `warehouse_zones.zone_code` is NOT NULL; `description`, `sort_order` nullable.  
- `warehouse_maps.width/height/grid_size/is_default` NOT NULL with defaults (2000/1200/20/false).  
- `locations.zone_id` FK `ON DELETE SET NULL` to `warehouse_zones(id)`.  
- `location_capacity_cache.location_id` PK and FK to `locations(id)`.
- `length_in`, `width_in`, `usable_height_in` have CHECK `> 0` when present.
- No explicit heatmap-specific check constraint on utilization percent found in map RPC layer.
- **Evidence (Confirmed):** migration definitions cited above.

### Occupancy percentage: stored vs derived
- `location_capacity_cache.utilization_pct` is stored cache output for suggestion engine flow.
- Heatmap zone/location utilization in viewer is **derived in RPC SELECT** (rounded percent), not read from `location_capacity_cache` in map RPCs.
- **Evidence (Confirmed):** `supabase/migrations/20260212160000_location_capacity_suggestion_engine.sql`; `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260219150000_hmv_p1_zone_location_capacity_rpc.sql`.

## 4) DR-4 CAPACITY CALCULATION LOGIC

### Where capacity % is calculated
- **Heat map zone/location percent:** calculated server-side in SQL RPCs (`ROUND((used/capacity)*100,2)`), not persisted by heatmap page.
- **Suggestion-engine cache percent:** maintained in `location_capacity_cache` via trigger functions using `items.size` and `locations.capacity_cuft`.
- **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260219150000_hmv_p1_zone_location_capacity_rpc.sql`; `supabase/migrations/20260212160000_location_capacity_suggestion_engine.sql`.

### Aggregation source
- Heatmap RPCs aggregate from:
  - `inventory_units.unit_cu_ft`
  - `containers.footprint_cu_ft` + container contents logic
  - `locations.capacity_cu_ft/capacity_cuft`
- **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260219150000_hmv_p1_zone_location_capacity_rpc.sql`; `supabase/migrations/20260219123000_hmv_p1_map_location_capacity_rpc.sql`.

### Released / quarantined / disposed handling
- Explicit status filters (`released`, `quarantined`, `disposed`) are **not visible** in heatmap RPC SQL excerpts; they scope by tenant + location joins and not by item status.
- **Status:** `[Unverified]` whether excluded upstream via `inventory_units` production rules.

## 5) DR-5 HEAT MAP INTENSITY MODEL

### Color scale mapping
- Static bucket mapping in client utility:
  - null/non-finite → gray (`#e5e7eb`)
  - `>100` → dark red (`#7f1d1d`)
  - `>=80` → red (`#ef4444`)
  - `>=50` → amber (`#f59e0b`)
  - else green (`#22c55e`)
- **Evidence (Confirmed):** `src/lib/heatMapUtils.ts` (`getHeatFill`).

### Thresholds/gradient/config/overrides
- Breakpoints are hardcoded buckets; no gradient interpolation.
- No user-configurable threshold settings found in heatmap page/hook path.
- No user override logic for color mapping found.
- **Evidence (Confirmed):** `src/lib/heatMapUtils.ts`; `src/pages/WarehouseHeatMap.tsx` legend labels.

## 6) DR-6 LOCATION GROUPING / ROW LABELING

### Zone grouping logic
- Persisted zone assignment uses `locations.zone_id` and map nodes (`warehouse_map_nodes.zone_id`).
- **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `src/hooks/useLocations.ts`; `src/hooks/useWarehouseMapNodes.ts`.

### Row naming / aliasing
- Zone alias text and group labels are map-node metadata (`label`, `group_label`) in map builder.
- Heatmap view renders `zone_code` text from zone-capacity rows; alias rendering in heatmap path is not present.
- **Evidence (Confirmed):** `src/pages/WarehouseMapBuilder.tsx` (`pendingNodeAliasUpdates`, `group_label` updates); `src/pages/WarehouseHeatMap.tsx` (zone label source uses `zone_code`).

### Visual-only vs persisted
- `group_label` and node `label` are persisted in DB.
- UI toggle “Zone Labels” controls visual display fallback behavior in builder, but heatmap route itself displays `zone_code` from RPC row.
- **Evidence (Confirmed):** `supabase/migrations/20260219160000_hmv_p1_map_node_group_labels.sql`; `src/pages/WarehouseMapBuilder.tsx`; `src/pages/WarehouseHeatMap.tsx`.

### Tenant scoping
- Zone/map/node objects include tenant-aware RLS policies and warehouse-access checks.
- **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql` (RLS + policy blocks; access checks in RPC).

## 9) DR-9 TENANT ISOLATION

### Static proof attempts
- `warehouse_maps`, `warehouse_zones`, `warehouse_map_nodes` have RLS policies tied to `user_tenant_id()`.
- Heatmap RPCs enforce:
  - map belongs to tenant (`wm.tenant_id = user_tenant_id()`)
  - warehouse access check via `user_has_warehouse_access(auth.uid(), v_warehouse_id)`
- `location_capacity_cache` RLS scopes via `locations.warehouse_id -> warehouses.tenant_id`.
- **Evidence (Confirmed):** `supabase/migrations/20260219120000_hmv_p1_warehouse_maps_zones.sql`; `supabase/migrations/20260212160000_location_capacity_suggestion_engine.sql`.

### Residual uncertainty
- Full end-to-end proof for every dependent table policy in aggregation CTEs (`inventory_units`, `containers`, `tenant_settings`) requires exhaustive policy audit beyond this slice.
- **Status:** `[Unverified]` for complete transitive RLS proof.
