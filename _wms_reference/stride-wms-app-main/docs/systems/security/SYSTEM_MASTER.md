## 1. FUNCTIONAL SCOPE (DR-1 SECURITY ARCHITECTURE OVERVIEW)

- Security posture is layered: route/UI role gates (`ProtectedRoute`, `RequireRole`) + database RLS + RPC/edge checks. **[Confirmed]**
- Trust boundary: browser client is untrusted; server-side authority comes from Postgres RLS policies/functions and service-role edge paths. **[Confirmed]**
- Multi-tenant assumption is pervasive (`tenant_id` on core domain tables; policy predicates frequently use `public.user_tenant_id()` or `users.tenant_id`). **[Confirmed]**
- Frontend guards are present but are not authoritative by themselves. **[Confirmed]**

Evidence:
- `src/components/ProtectedRoute.tsx`, `src/components/RequireRole.tsx`, `src/App.tsx`
- `supabase/migrations/20260208000000_client_portal_rls_policies.sql`
- `supabase/migrations/20260202200000_admin_dev_system_role.sql`

## 2. TENANT BOUNDARY MODEL (DR-2)

- Tenant predicates are enforced broadly in policies (`tenant_id = public.user_tenant_id()` and `tenant_id IN (SELECT tenant_id FROM public.users WHERE id = auth.uid())`). **[Confirmed]**
- Cross-domain tables (billing, claims, shipments, tasks, account activity) contain explicit tenant-bound RLS examples. **[Confirmed]**
- Static scan found 174 created public tables and at least one RLS enable statement for all discovered created tables in migrations. **[Confirmed]**
- Legacy tables created outside current migration history are not provably covered in this static pass. **[Unverified]**

## 3. ACCOUNT BOUNDARY MODEL (DR-3)

- Client portal account boundary is server-enforced via `public.client_portal_account_id()` and account-scoped SELECT policies. **[Confirmed]**
- Staff tenant-wide access is separated from client account-scoped access by `public.is_client_user()` predicates in policy definitions. **[Confirmed]**
- Client-side filters exist in hooks/pages, but account isolation is not solely UI-based where portal RLS is applied. **[Confirmed]**
- Account-scoped write controls for all portal-capable entities were not fully enumerated in this pass. **[Unverified]**

Evidence:

## 4. ROLE-BASED ACCESS MODEL (DR-4)

- Role claims are loaded from DB join (`user_roles -> roles`) in app hooks (`usePermissions`, `useCurrentUserRole`). **[Confirmed]**

- DB also enforces role semantics via helper functions/policies (`current_user_is_admin_dev`, `has_role(...)`). **[Confirmed]**

## 6. RPC SECURITY SURFACE (DR-6)

- Migration scan identified a large function surface (`CREATE OR REPLACE FUNCTION public.*`), with a majority using `SECURITY DEFINER`. **[Confirmed]**
- Positive pattern exists: some definer functions pin `search_path = public` (e.g., `user_has_warehouse_access`, client portal helpers). **[Confirmed]**
- Risk pattern exists: numerous `SECURITY DEFINER` functions in sampled files omit explicit `SET search_path`. **[Risk]**
- Function-by-function guard validation (tenant checks, role checks, grants) is not complete for all RPCs. **[Unverified]**

Evidence:
- `supabase/migrations/20260224100000_put_away_config_rules.sql`

## 7. EDGE FUNCTION SECURITY SURFACE (DR-7)

- Many edge functions use `SUPABASE_SERVICE_ROLE_KEY` and then perform explicit auth header/JWT checks for caller identity before privileged operations. **[Confirmed]**
- Service-role client usage is common for operational tasks (Stripe, SMS, email, admin ops), increasing blast radius if guard logic is weak. **[Risk]**
- Specific anomaly: `qa-runner` creates a user client with `supabaseServiceKey.replace('service_role', '')`, a non-standard auth pattern that warrants hardening review. **[Risk]**
- Downstream provider authorization semantics (Stripe/Twilio/Postmark/QBO) are static-only reviewed here. **[Unverified]**

Evidence:
- `supabase/functions/send-email/index.ts`
- `supabase/functions/send-sms/index.ts`
- `supabase/functions/create-stripe-checkout-session/index.ts`
- `supabase/functions/qa-runner/index.ts`

## 9. DIRECT INSERT / BYPASS RISK (DR-9)

- App includes direct table inserts from client flows (e.g., claims creation) alongside RPC-based flows; this is acceptable only if RLS and constraints are strict. **[Confirmed]**
- Mixed direct insert + RPC patterns increase drift risk where validation logic exists in only one path. **[Risk]**
- Comprehensive bypass mapping across all domain writes was not exhaustively completed. **[Unverified]**

## 12. CROSS-SYSTEM AUTHORIZATION DRIFT (DR-12)

- Billing, shipments, tasks, claims, and portal each show tenant/account constraints, but enforcement style differs (direct table CRUD vs guarded RPC vs edge+service-role). **[Confirmed]**
- Drift risk is elevated where the same domain operation appears in both direct writes and RPC wrappers. **[Risk]**
- No canonical centralized authorization policy registry found in-repo to enforce consistency across these systems. **[Risk]**

Evidence:
- Billing RLS/function sample: `supabase/migrations/20260124044004_453dfddc-767a-4298-b662-fda6ce66a0df.sql`
- Portal account RLS: `supabase/migrations/20260208000000_client_portal_rls_policies.sql`
- Activity tracking: `supabase/migrations/20260206210000_shipment_task_account_activity.sql`
- Claims/client writes: `src/pages/ClientClaims.tsx`, `src/hooks/useClaims.ts`

## 13. KNOWN SECURITY GAPS (DR-13)

- Incomplete static proof of full CRUD RLS coverage for every historical table/policy permutation. **[Unverified]**
- High `SECURITY DEFINER` volume with inconsistent explicit search_path pinning. **[Risk]**
- Service-role heavy edge architecture increases impact of any missed authorization check. **[Risk]**
- Mixed direct-write and RPC-write patterns can introduce validation bypass drift over time. **[Risk]**
- Runtime protections (platform secrets, edge config, network controls) were not verifiable in static repository scope. **[Unverified]**

---

## 14. HARDENING RECOMMENDATIONS (DR-14, DOC-ONLY)

1. Enforce a migration lint rule: every `SECURITY DEFINER` function must include `SET search_path = public` (or an explicit approved schema list). **[Recommended]**
2. Build and maintain a generated security inventory artifact per release: table → RLS enabled → CRUD policies present → predicate class (tenant/account/role). **[Recommended]**
3. Standardize privileged edge middleware (auth parsing, tenant resolution, role checks, audit logging) and require all service-role functions to use it. **[Recommended]**
4. Reduce direct table writes for high-risk flows (claims, billing-affecting events, admin operations) in favor of audited RPC entrypoints. **[Recommended]**
5. Add policy tests (static SQL assertions in CI) for “client user cannot cross-account read/write” and “staff cannot cross-tenant write.” **[Recommended]**
6. Add a secure-function checklist for PRs: grants, invoker/definer rationale, search_path, tenant/account predicates, role checks, and audit trail. **[Recommended]**

### Suggested Prompt Revisions (for future Phase-14 runs)

- Require an explicit machine-generated appendix: `SECURITY DEFINER` functions lacking `SET search_path`.  
- Require a complete table-level matrix (all public tables) with columns: `has_tenant_id`, `has_account_id`, `RLS_enabled`, `SELECT/INSERT/UPDATE/DELETE policy presence`.  
- Require a dedicated section listing all edge functions by auth model: `public`, `JWT-required`, `service-role internal`, with risk grade.
