## 1. FUNCTIONAL SCOPE (DR-1)

- **Primary purpose:** Capture, assess, settle, and track claims (liability + shipping damage assistance) with item, shipment, attachment, audit, and billing touchpoints. **[Confirmed]**
- **Explicit non-goals:** Billing core pricing internals and generalized inventory mutation internals are out of scope for this document (only claim side-effects are included). **[Confirmed]**
- **Cross-system touchpoints:**
  - **Inventory:** optional `item_id` and multi-item `claim_items` links. **[Confirmed]**
  - **Shipments:** optional `shipment_id`, shipment-derived coverage support in claim item creation. **[Confirmed]**
  - **Billing:** `createEventRaw` for `claim_assistance` and `account_credits` issuance path. **[Confirmed]**
  - **Client Portal:** `useClientClaims`, `useClientClaimDetail`, and tokenized acceptance route `/claim/accept/:token`. **[Confirmed]**
  - **Documents/Media:** `claim_attachments` + Supabase Storage bucket `claims`. **[Confirmed]**

## 2. ARCHITECTURAL WIRING (DR-2)

### Canonical Trace A — Staff/Internal Claim Creation

`/claims` (admin route) → `Claims.tsx` + `ClaimCreateDialog` → `useClaims.createClaim()` → `supabase.from('claims').insert(...)` → optional `claim_items` inserts + optional `claim_assistance` billing event + `claim_audit` insert + queued alerts → refetch + toast. **[Confirmed]**

### Canonical Trace B — Client Portal Claim Creation

`/client/claims` (client_user route) → `ClientClaims.tsx` `handleCreateClaim()` → `supabase.from('claims').insert(...)` (with `client_initiated=true`) → query invalidation + toast. **[Confirmed]**

### Acceptance/Settlement Trace

Public route `/claim/accept/:token` → `ClaimAcceptance.tsx` RPCs `get_claim_by_acceptance_token`, `get_claim_items_by_acceptance_token`, `accept_claim_settlement` / `decline_claim_settlement` → claim status and acceptance logs/audit updated in DB functions. **[Confirmed]**

| `claims` | Claim header/lifecycle | `tenant_id`, `account_id`, `item_id`, `shipment_id`, `claim_type`, `claim_category`, `status`, settlement/acceptance/SLA columns | Enabled + tenant/client policies present | Claims |
| `claim_items` | Per-item valuation + payout logic | `claim_id`, `item_id`, `requested_amount`, `approved_amount`, POP and valuation fields | Client SELECT policy exists; base policy definition location for full CRUD is partially outside inspected files | Claims |

### Status / Type Constraints

- Earlier claim status constraints evolved from legacy (`open`, `investigating`, etc.) to `claims_status_check` (`initiated`,`under_review`,`denied`,`approved`,`credited`,`paid`,`closed`) plus acceptance workflow comment/status additions (`pending_acceptance`,`accepted`,`declined`,`pending_approval`). **[Risk]** potential schema/app drift; see DR-5.
- `claim_type` constraint in migration includes values such as `shipping_damage`, `manufacture_defect`, `handling_damage`, `property_damage`, `lost_item`. **[Confirmed]**
- `claim_category` adds `liability` vs `shipping_damage`. **[Confirmed]**

### Timestamps / Timezone

- Claims and related tables primarily use `TIMESTAMPTZ` with `now()` defaults in migrations.
- UI writes use ISO timestamps via `new Date().toISOString()` for some updates.
**[Confirmed]**

## 5. CLAIM STATUS MACHINE (DR-5)

### Enumerated Statuses (Source Aggregation)

- **In app type labels:** `initiated`, `under_review`, `pending_approval`, `pending_acceptance`, `accepted`, `declined`, `denied`, `approved`, `credited`, `paid`, `closed`. **[Confirmed]**
- **In migration `claims_status_check`:** `initiated`, `under_review`, `denied`, `approved`, `credited`, `paid`, `closed`. **[Confirmed]**
- **In acceptance migration comment + functions:** includes `pending_acceptance`, `accepted`, `declined`, and references `pending_approval`. **[Confirmed]**

### Transition Highlights

- `createClaim` inserts status `initiated`. **[Confirmed]**
- `send_claim_for_acceptance` function sets status `pending_acceptance` and stores pre-status. **[Confirmed]**
- `accept_claim_settlement` and `decline_claim_settlement` functions set accepted/declined outcomes. **[Confirmed]**
- `updateClaimStatus` handles direct status writes and sets `resolved_at/resolved_by` for approved/credited/paid. **[Confirmed]**

### Transition Side Effects

- `claim_audit` records lifecycle actions. **[Confirmed]**
- Alert queue functions are invoked on filed/status changed/approved/denied paths. **[Confirmed]**
- Shipping-damage category may emit billing event `claim_assistance`. **[Confirmed]**

### Divergence Assessment

- **[Risk]** Status vocabulary appears broader in app + RPC logic than in one explicit DB check constraint migration (`claims_status_check`). This may be resolved in unseen migrations, but static proof was not found in inspected subset.

## 9. MEDIA / EVIDENCE STORAGE (DR-9)

- Storage bucket `claims` is created as non-public in migration.
- Storage path convention in hooks/pages: `${tenant_id}/${claim_id}/${uuid.ext}`.
- Storage object policies enforce first path segment matching user tenant for SELECT/INSERT/DELETE on bucket `claims`.
- `claim_attachments` rows store file metadata and `is_public`; client detail query filters public attachments only.
- No explicit signed-URL calls (`createSignedUrl`) found in inspected claims paths; direct object path usage is predominant. **[Unverified]** whether other claim components use signed URLs.
- Retention/deletion: delete path removes storage object and attachment row in `useClaims.deleteAttachment`. **[Confirmed]**
