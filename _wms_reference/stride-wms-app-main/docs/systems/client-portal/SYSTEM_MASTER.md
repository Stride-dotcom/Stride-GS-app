## DR-1 FUNCTIONAL SCOPE
- **Purpose:** Authenticated customer-facing portal for account-scoped visibility and request submission across items, shipments, quotes, claims, and task/outbound/inbound requests. [Live]
- **Non-goals:** Internal staff console workflows (billing operations, warehouse map internals, admin ops) are wired outside `/client/*` route space. [Live]
- **Cross-system touchpoints:**
  - **Shipments:** list/detail plus inbound/outbound create.
  - **Tasks:** client task requests and split/pending_review metadata handoff.
  - **Claims:** claim creation + attachment upload + claim detail acceptance link.
  - **Quotes:** client accept/decline repair quotes.
  - **Alerts:** split-required/manual-review alert queuing.
  - **Documents/Media:** claim bucket uploads and shipment photo/document rendering.
  - **Billing/Invoices:** no dedicated `/client/invoices` route found in current router. [N/A]

## DR-2 ROUTE MAP & ENTRYPOINTS
Portal routes are mounted in `src/App.tsx` and guarded by `ProtectedRoute + RequireRole('client_user')` except login.

- `/client/login` → `ClientLogin` (public entry)
- `/client` → `ClientDashboard`
- `/client/items` → `ClientItems`
- `/client/quotes` → `ClientQuotes`
- `/client/shipments` → `ClientShipments`
- `/client/shipments/new` → `ClientInboundCreate` (subscription-gated)
- `/client/shipments/outbound/new` → `ClientOutboundCreate` (subscription-gated)
- `/client/shipments/:id` → `ClientShipmentDetail`
- `/client/tasks/new` → `ClientTaskCreate`
- `/client/claims` → `ClientClaims`

Portal layout/nav wrapper: `src/components/client-portal/ClientPortalLayout.tsx` with menu entries Dashboard, My Items, Shipments, Quotes, Claims.

## DR-3 AUTH / ROLES / SESSION MODEL
- **Session model:** `AuthContext` listens to Supabase auth state/session and fetches profile from `users` table. [Live]
- **Role checks:** `RequireRole` uses `usePermissions().hasRole('client_user')`; `usePermissions` prefers RPC `get_my_roles` and falls back to `user_roles` join. [Live]
- **Portal-specific access check:** `ClientLogin` additionally requires active row in `client_portal_users` for authenticated user (`auth_user_id` + `is_active=true`), otherwise signs user out. [Live]
- **Guarding behavior:** `ProtectedRoute` redirects unauthenticated users to `/auth` (staff login path), not `/client/login`. [Risk: UX mismatch for deep-linked client pages]
- **Client vs staff differentiation:** frontend role gate uses RBAC role `client_user`; data context uses `client_portal_users` account binding. [Partial: dual-source model]

## DR-4 ACCOUNT SCOPING MODEL (CRITICAL)
### How “my account_id” is derived
- `useClientPortalContext` queries `client_portal_users` by `auth_user_id` and reads `account_id`.
- All core portal list hooks apply explicit `.eq('account_id', portalUser.account_id)` filters.

### Enforcement posture classification
- **Observed posture:** **RLS + client-side filters + frontend role checks**.
  - RLS layer: `client_portal_account_id()` + `is_client_user()` policies for SELECT on items/shipments/claims/etc.
  - Visibility layer: portal hooks add account filters in queries.
  - Frontend role layer: `RequireRole('client_user')`.

### Ordering (normal flow)
1. User authenticates via Supabase session.
2. Route guard requires authenticated user + `client_user` role.
3. Portal context resolves account from `client_portal_users`.
4. UI queries apply account filter.
5. DB RLS enforces tenant/account row visibility.

### Failure modes
- **If UI filters fail or are removed:** RLS should still constrain SELECT for covered tables. [Live for tables with client policies]
- **If RBAC role assignment is wrong but `client_portal_users` exists:** user can still pass login but route guards may block if role absent. [Partial]
- **If INSERT/UPDATE policies are tenant-only (not account-scoped):** client could potentially write cross-account data within tenant if client-controlled payload includes another account_id. [Risk]
  - Examples: base `shipments` and `tasks` insert policies are tenant-scoped; no account-scoped client INSERT policy found in reviewed migrations.

### Explicit separation
- **Enforcement layer (hard auth):** Supabase RLS policies + role checks on routes.
- **Visibility layer (UI/query):** account-filtered selects in portal hooks/pages.
- **Assumption layer:** assumes `client_portal_users.account_id` is trustworthy and aligned with `client_user` role; assumes write paths are protected by stricter downstream constraints.

## DR-5 DATA MODEL (PORTAL-RELEVANT SLICE)
- **`client_portal_users`**: tenant/account/auth-user mapping, active flag, last login, unique `(tenant_id, email)`; RLS enabled in migration.
- **`shipments`**: portal reads shipment header fields and writes inbound/outbound drafts/requests; includes `metadata` for `pending_review` / `split_required`.
- **`tasks`**: portal writes task requests and split-task metadata (`split_workflow`, `split_workflow_items`).
- **`claims`**: portal reads/writes claim records with client-facing status and acceptance fields.
- **`claim_attachments` + storage bucket `claims`**: portal uploads files, marks `is_public=true`, and reads public attachments.
- **`repair_quotes`**: portal lists and updates quote status (`accepted`/`declined`).
- **`invoices`**: table exists in type contract, but no dedicated client route/surface detected in this phase. [N/A]

Authority application: migration definitions and RLS policies were prioritized, then typed contract (`src/integrations/supabase/types.ts`), then UI behavior.

## DR-6 CLIENT SHIPMENTS (VIEW/CREATE) FLOWS
- **View shipments:** `ClientShipments` → `useClientShipments` → `shipments SELECT account_id filter` → status rendering with `pending_review`/`split_required` metadata badges. [Live]
- **Shipment detail:** `ClientShipmentDetail` → shipment SELECT by `id + account_id` + item join query → detail UI + shipment photos/documents/notes sections. [Live]
- **Create inbound:** `ClientInboundCreate` → insert/update draft shipment + optional `create_unified_note` RPC + insert `items` + insert `shipment_items` + toast + navigate. [Live]
- **Create outbound:** `ClientOutboundCreate` → insert/update shipment + insert shipment_items + split workflow (create split tasks OR mark pending_review metadata) + alert queue + toast + navigate. [Live]
- **Release by client:** no dedicated direct release action found beyond outbound request submission. [N/A]

## DR-7 CLIENT TASK REQUEST FLOWS
- `ClientTaskCreate` creates `tasks` row, links `task_items`, and conditionally:
  - creates Split tasks with `split_workflow` metadata + alerts, or
  - marks origin task `pending_review` with `split_workflow_items` + manual review alert.
- `ClientOutboundCreate` mirrors the same split metadata concepts on shipments.
- `pending_review` lifecycle is visible in client shipment/task-facing metadata labels and intended staff handoff via alert queue.
- Cross-reference: aligns conceptually with Tasks SYSTEM_MASTER split/pending_review model. [Live/Partial: static-only verification]

## DR-8 CLIENT CLAIMS FLOWS
- **Create claim:** `ClientClaims.handleCreateClaim` inserts into `claims` with `client_initiated=true` and generated claim number.
- **View claims:** hooks query account-scoped claims and mask payout fields until `sent_for_acceptance_at`.
- **Attachments:** uploads files to storage bucket `claims` path `{tenant_id}/{claim_id}/{uuid.ext}`, then inserts `claim_attachments` and `claim_audit` records.
- **Settlement acceptance:** detail screen links to tokenized acceptance route (`/claim/accept/:token`) when pending acceptance.
- Cross-reference: consistent with Claims SYSTEM_MASTER portal-facing claim lifecycle.
