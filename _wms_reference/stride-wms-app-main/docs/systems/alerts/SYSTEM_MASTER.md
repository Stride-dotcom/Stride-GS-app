## 2. ARCHITECTURAL WIRING (DR-2)

### Canonical trace (email + in-app)
System event in UI/hook/service → `queueAlert()` write to `alert_queue` (`status='pending'`) → immediate `supabase.functions.invoke('send-alerts')` best-effort trigger → `send-alerts` fetches pending queue rows, resolves recipients/templates/channel toggles, sends email provider request, optionally inserts `in_app_notifications`, then updates `alert_queue.status` to `sent|failed|skipped`. **Confirmed**.

### Variants
1. **Disabled alert/email channel:** marks queue row `skipped`. **Confirmed**.
2. **No resolved recipients:** marks queue row `failed`. **Confirmed**.
3. **In-app generation errors:** logged, but email path continues and queue can still be marked sent. **Confirmed**.
4. **Provider send-log compatibility fallback:** retries update without provider fields if schema mismatch occurs. **Confirmed**.

Evidence: `src/lib/alertQueue.ts` (queue + invoke), `supabase/functions/send-alerts/index.ts` (pending query, channel checks, in-app inserts, sent/failed/skipped updates).

## 3. DATA MODEL (DR-3)

### Primary tables observed
| Table | Purpose | Key fields observed | Notes |
|---|---|---|---|
| `alert_queue` | queued outbound alert jobs | `tenant_id`, `alert_type`, `entity_type`, `entity_id`, `recipient_emails`, `status`, `sent_at`, `provider`, `provider_message_id`, `fallback_used`, `error_message` | queue source of truth for sender function |
| `communication_alerts` | per-tenant trigger + channel policy | `tenant_id`, `trigger_event`, `is_enabled`, `channels` | gating for email/in-app channel use |
| `communication_templates` | per-alert channel templates | `channel`, `subject_template`, `body_template`, `in_app_recipients` | includes `in_app` channel |
| `in_app_notifications` | per-user in-app delivery + read/archive state | `user_id`, `tenant_id`, `is_read`, `read_at`, `is_archived`, `archived_at`, `alert_queue_id` | read/unread lifecycle |
| `notification_events` | event-log style table in DB types | `alert_type_id`, `entity_type`, `entity_id`, `event_payload` | present in schema types; not central in queue sender path |

### Enumerations / constraints observed
- `communication_templates.channel` constrained to `('email','sms','in_app')`. **Confirmed**.
- `alert_queue.provider` constrained to `('resend','postmark')` or null. **Confirmed**.
- `in_app_notifications` has direct FK to `alert_queue` via `alert_queue_id`. **Confirmed**.

### RLS/policy evidence
- In-app + message notifications include RLS and user-scoped access/mutations in migrations (`in_app_notifications_user_access`, auth.uid()-scoped mark/archive helpers). **Confirmed**.
- Full alert_queue RLS policy proof is not fully established in inspected subset. **[Unverified]**.
