## DR-1 FUNCTIONAL SCOPE (PRICING SUBSYSTEM)

- This subsystem is the tenant-facing billing configuration layer for service/charge definitions and rate rules (`charge_types` + `pricing_rules`), with legacy compatibility through `service_events`. [Confirmed]
- It supports service metadata (trigger, category, taxable, scan/flag behavior), class-based/flat rules, and tenant bootstrap seeding for “core” and “starter” catalogs. [Confirmed]
- Explicit non-goal: direct invoice generation/payment processing internals remain in Billing system authority. [Confirmed]
- Cross-system touchpoints observed in code: Billing (rate lookup), Tasks (task billing service code resolution), Shipments/Receiving (billing preview), Quotes (price-list consumption), Client Portal ([N/A] direct pricing editor surface not found). [Partial]

### Canonical traces

`Settings route -> ServiceRatesConsole -> PriceListTab -> useChargeTypesWithRules -> supabase.from('charge_types'/'pricing_rules') -> table render + filter/export UI`.

2) **Edit charge type / pricing rule**  
`PriceListTab -> AddServiceForm(edit mode) -> useChargeTypes.updateChargeType + delete/recreate pricing_rules -> toast + refetch`.

`PriceListTab -> ServiceRateImportDialog -> parseFileToRows + row validation -> upsert charge_types + pricing_rules -> progress + result badges`.

`PriceListTab -> ExcelJS workbook builder -> export current filtered list OR template + instruction sheet`.

## DR-14 EXTENSION GUIDE (HOW TO ADD NEW CHARGES SAFELY)

1. Add/verify category in `service_categories` (or use existing system category).  
2. Create `charge_types` row with canonical `charge_code`, trigger, tax, and scan/flag fields.  
3. Add at least one `pricing_rules` row (class-specific and/or `is_default=true` fallback).  
4. If task-driven, map task type via `primary_service_code` (retain `default_service_code` fallback compatibility).  
5. Validate preview/read paths by tracing `getRateFromPriceList`/`getEffectiveRate` use in Tasks + Shipments/Receiving flows.  
6. If onboarding defaults needed, extend seeding functions (`seed_core_charge_types` / `seed_starter_charge_types` / template RPCs).  
7. Re-check RLS posture for any newly introduced pricing tables/RPCs.
