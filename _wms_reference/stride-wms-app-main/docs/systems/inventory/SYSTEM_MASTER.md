## 1) DR-1 FUNCTIONAL SCOPE

### Purpose
Inventory is the canonical system-of-record for warehouse item identity, lifecycle state, physical location linkage, shipment linkage, and storage/coverage-relevant fields used by billing and claims flows.

### Explicit Non-Goals
- Warehouse map geometry internals.
- Capacity heatmap/suggestion engine internals.
- Stocktake operational flows.
- Reporting-domain analytics.

### Cross-System Touchpoints
- **Shipments / Receiving:** item creation/update on intake; shipment-to-item linking via `items.receiving_shipment_id` and `shipment_items.item_id`.
- **Tasks:** task completion updates item task-status fields (`assembly_status`, `inspection_status`, `repair_status`) and may release/dispose items depending on task type.
- **Billing:** storage eligibility uses receive/release semantics (`received_date`/`released_date` in RPC path and `received_at`/`released_at` in UI preview path); coverage billing events can be emitted from item declared value updates.
- **Claims:** item declared value + coverage fields are consumed by claim/coverage logic.
- **Warehouse Map:** only via location FK linkage and receiving-location resolution RPCs (no map geometry behavior documented here).

## 2) DR-2 ARCHITECTURAL WIRING (Canonical Full-Stack Traces)

### A. Item Creation (Receiving)
1. Stage 2 receiving completion path resolves receiving location (`rpc_resolve_receiving_location`), computes effective receiving dock/location, and finalizes shipment state.
2. Receiving session logic checks if a `shipment_item` already has an `item_id`:
   - if yes: update existing `items` row (`status='active'`, `received_at`, location fields, declared value carry-over);
   - if no: insert new `items` row with `receiving_shipment_id`, `status='active'`, `received_at`, and optional declared value.
3. `shipment_items` row is updated to received status and linked to item.
4. Optional task fan-out from receiving: auto-create Inspection / Assembly tasks and link via `task_items`.

Variant notes:
- Stage 2 also has a safety-net RPC (`rpc_assign_receiving_location_for_shipment`) to assign location for shipment-linked items lacking current location.
- Legacy/default precedence for receiving location uses warehouse default, account default, then first receiving-type location.

### B. Item Status Change (Task completion or Shipment/Release path)
1. Task completion in `useTasks`:
   - For standard Assembly/Inspection/Repair tasks: mapped task status updates corresponding item field.
   - For Will Call: task completion sets linked items `status='released'`.
   - For Disposal: task completion sets linked items `status='disposed'` and `deleted_at`.
2. Release dialogs can also update item release state directly (`status='released'`, `released_at`) and create outbound shipment rows.
3. DB trigger path exists for location-driven release: when `current_location_id` moves to a location typed `release`, trigger sets `status='released'`, `released_at=now()`.

Variant notes:
- Multiple pathways can release items (task flow, release dialogs, location trigger), so release-side behavior is distributed across app + DB trigger layers.

### C. Item Location Change
1. UI/hook updates `items.current_location_id` (and sometimes `location_id`) directly for item moves / put-away.
2. DB trigger `create_movement_on_location_change` writes `movements` audit rows whenever location changes.
3. Batch movement function `batch_move_items(...)` supports mass reassignment and movement logging.

Variant notes:
- Some flows use `current_location_id` as operational location while `location_id` remains as an additional linkage field.

### D. Item Archival/Release
1. Disposal task completion marks item `status='disposed'` + soft-delete timestamp.
2. Release flows (Quick Release / task completion / trigger) set released semantics.
3. Storage billing logic excludes released items by date-window checks (`released_date` or `released_at` depending path).

## 3) DR-3 DATA MODEL

## Source-of-Truth Rule
For enums/statuses/schema claims, **DB migrations/schema are authoritative over `types.ts`/UI assumptions**. Any mismatch is tagged as `[Risk]` or `[Legacy]`.

### Core `items` Fields in Scope
- Identity/tenancy: `id`, `tenant_id`, `warehouse_id`, `account_id`, `item_code`.
- Lifecycle/status: `status`, `deleted_at`, `received_at`, `released_at`, `received_date`, `released_date`.
- Shipment linkage: `receiving_shipment_id` (FK to shipments).
- Location linkage: `current_location_id`, `location_id`.
- Class/behavior linkage: `class_id`, `quantity`, `metadata`, `condition`.
- Coverage/declared value linkage: `declared_value`, `coverage_type`, `coverage_rate`, `coverage_deductible`, `coverage_source`, `coverage_selected_at/by`.
- Task-related status fields: `assembly_status`, `inspection_status`, `repair_status`.
- Damage/quarantine-adjacent: `has_damage` plus indicator flags via `item_flags` records.
- Storage fields: `last_storage_invoiced_through`, `received_date`, `released_date`, plus timestamp equivalents.

### Related Objects
- `shipment_items`: links expected/actual shipment rows to `items` (`item_id`) and tracks receive/release status.
- `movements`: written on location change to track inventory movement history.
- `item_flags`: used for indicator-based flags (including quarantine indicator patterns).
- `coverage_history`: audits coverage and declared-value changes.

### Nullability / Status / Timezone Notes
- `received_at` / `released_at` are nullable `timestamptz` (timestamp with timezone semantics).
- `received_date` / `released_date` are date derivations used by storage billing RPC path.
- [Legacy] Historical enum declaration includes `('pending','active','released','disposed','lost')`, but active code paths also use additional values (e.g., `stored`) in split workflow insert logic.
- [Risk] Item status constraints are not clearly centralized in one visible migration in this slice; status usage is effectively multi-source (UI + SQL functions + historical enum declarations).

## 4) DR-4 ITEM STATUS MACHINE

### Enumerated/Observed States
- `active`
- `released`
- `disposed` (applicable)
- `quarantined` (modeled more strongly via flags/units; item-row status usage is mixed)
- `damaged` (primarily represented by `has_damage` and related indicators)
- partial/intermediate states seen in code paths: `pending`, `stored`, `lost` [Legacy]/[Risk]

### Required Transitions & Trigger Sources
- **Receiving → Active**
  - Source: receiving UI/hook updates/inserts item with `status='active'`.
- **Active → Released**
  - Source: Will Call completion, Quick Release dialog, or DB location-release trigger.
- **Active → Disposed**
  - Source: Disposal task completion handler.
- **Active → Quarantined**
  - Source: primarily flag/exception logic (indicator + quarantine location/units), not a single canonical item status mutation in this slice.
- **Quarantined → Cleared**
  - Source: Repair completion path clears `has_damage` and removes quarantine indicator flags from `item_flags`.

## 5) DR-5 SHIPMENT INTEGRATION

- Item-to-shipment attachment occurs through:
  - `items.receiving_shipment_id` (inbound ownership lineage), and
  - `shipment_items.item_id` (join table for expected/actual shipment lines).
- Release creation creates outbound `shipments` and `shipment_items`; quick-release can immediately set shipment-item status to released.
- Partial release handling exists at shipment-item granularity (`shipment_items.status` supports lifecycle values including partial in legacy enum declaration).
- RLS on `shipment_items` is enforced through tenant checks via parent `shipments`.

## 6) DR-6 TASK INTEGRATION

- Task type to item-field mapping:
  - `Assembly` → `assembly_status`
  - `Inspection` → `inspection_status`
  - `Repair` → `repair_status`
- Task status map writes item field values (`pending`, `in_progress`, `completed`, `unable_to_complete`).
- Task-linked UI surfaces can resolve sibling open work from `task_items` for the same item:
  - `TaskDetail` shows status-bar jump links for other `pending` / `in_progress` tasks sharing the current item(s),
  - `ItemDetail` routes single matching open-task shortcuts directly to `/tasks/:id`.
- Repair completion includes damage/quarantine clearing logic:
  - set `has_damage=false` on linked items,
  - remove quarantine indicator flags from `item_flags` where matching charge code,
  - log item activity events for audit trail.
- Special task types:
  - `Will Call` completion releases items,
  - `Disposal` completion disposes items.

## 7) DR-7 STORAGE ELIGIBILITY LOGIC

### Canonical DB RPC Path
`generate_storage_for_date(p_date)` determines storage eligibility from `items` + `accounts`:
- Include item when:
  - `status='active'`,
  - `received_date <= p_date`,
  - `released_date IS NULL OR released_date > p_date`.
- Free days rule: only create billing events when `((p_date - received_date)+1) > accounts.free_storage_days`.

### UI Billing Preview Path
`useBilling` computes active-period candidates using timestamp semantics:
- `received_at <= periodEnd`
- `released_at IS NULL OR released_at >= periodStart`
- Applies free storage days from tenant preferences with day-window arithmetic.

### Edge Cases
- Same-day receive/release behavior differs by strict inequality/date arithmetic path.
- Null receive/release fields exclude eligibility in DB path (requires `received_date` non-null).
- [Risk] RPC path uses account-level `free_storage_days`; UI preview path uses tenant preference `free_storage_days`.

Cross-reference: `docs/systems/billing/SYSTEM_MASTER.md`.

## 8) DR-8 LOCATION LINKAGE

- Item location fields in active use: `current_location_id` (operational) and `location_id` (additional linkage).
- Default receiving location logic:
  1. warehouse default receiving location,
  2. account default receiving location,
  3. first receiving-type location in warehouse.
- RPCs involved:
  - `rpc_resolve_receiving_location`
  - `rpc_assign_receiving_location_for_shipment`
- Reassignment logic:
  - direct item updates in UI/hook flows,
  - `batch_move_items` function for bulk moves,
  - movement audit via location-change trigger.
- RLS on location access is tenant-warehouse scoped through `locations` policies based on warehouse tenant ownership.

(No warehouse map geometry documented here by design.)

## 9) DR-9 SPLIT WORKFLOW LINKAGE

- `rpc_apply_grouped_item_split_off_leftover`:
  - reduces parent grouped quantity (`keep_qty`),
  - creates child items (`qty=1`) with derived item codes,
  - writes split metadata on children including:
    - `split_parent_item_id`
    - `split_parent_item_code`
    - `split_task_id`
- `rpc_complete_split_task` validates split invariants using DB truth:
  - child count and quantity checks,
  - parent linkage checks,
  - child code non-collision checks,
  - prunes `split_required_*` metadata on origin shipment/task.
- Parent/child linkage is metadata-based rather than a dedicated normalized parent_id FK column.

Cross-reference: task system split flow documentation.

## 10) DR-10 COVERAGE / DECLARED VALUE

- Declared value fields:
  - `shipment_items.declared_value` (pre-receiving capture),
  - `items.declared_value` (inventory-level persisted value).
- Coverage interactions:
  - `rpc_apply_shipment_coverage` applies paid coverage at shipment scope and cascades per-item billing deltas.
  - `rpc_update_item_declared_value` updates item DV, enforces min threshold, resolves rate, and emits coverage billing delta when applicable.
- Billing linkage:
  - writes `billing_events` with `event_type='coverage'` and `charge_type='handling_coverage'` when delta exists.
  - writes `coverage_history` audit rows for DV and coverage changes.

Cross-reference: `docs/systems/billing/SYSTEM_MASTER.md`.
