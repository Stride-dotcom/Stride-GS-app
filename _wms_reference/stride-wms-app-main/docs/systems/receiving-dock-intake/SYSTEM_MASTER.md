## DR-1 FUNCTIONAL SCOPE

### Purpose
Receiving / Dock Intake runs inbound intake in two phases:
1) Stage 1 dock capture updates shipment intake fields and marks `inbound_status='stage1_complete'`. Evidence: `src/components/receiving/Stage1DockIntake.tsx:764-790`.
2) Stage 2 receiving closes intake, sets `status='received'`, `inbound_status='closed'`, and timestamps. Evidence: `src/components/receiving/Stage2DetailedReceiving.tsx:1573-1583`.

### Explicit Non-goals
- Core billing pricing engine internals are outside this system; only preview invocation is visible here. Evidence: `src/components/receiving/Stage2DetailedReceiving.tsx:1635-1638`.
- Routing internals beyond receiving route wiring are out-of-scope.

### Cross-system touchpoints
- Shipments + shipment_items writes: Stage 1 + Stage 2 flows. Evidence: `Stage1DockIntake.tsx:786-790`, `Stage2DetailedReceiving.tsx:1545-1550`, `1575-1583`.
- Inventory items creation/update on receiving. Evidence: `Stage2DetailedReceiving.tsx:1359-1364`, `1590-1599`.
- Billing preview only: `calculateShipmentBillingPreview(...)`. Evidence: `Stage2DetailedReceiving.tsx:1636`; `useReceivingSession.ts:213-217`.
- Tasks auto-create (inspection/assembly) in receiving session path. Evidence: `useReceivingSession.ts:429-488`.
- Claims/Coverage: auto-coverage RPC optional. Evidence: `useReceivingSession.ts:556-563`; migration `20260225120100_coverage_option_a_rpcs.sql:636-701`.
- Alerts: receiving/unidentified/coverage alerts queued. Evidence: `useReceivingSession.ts:514-553`, `567-580`; `Stage2DetailedReceiving.tsx:1697-1705`.

## DR-2 ARCHITECTURAL WIRING (Canonical traces)

### Trace A — Draft creation/resume path
`/incoming/dock-intake/:id` → `DockIntakeReceiving` → `ReceivingStageRouter` fetches shipment and renders Stage 1/2 → Stage 1 edits autosave to `shipments` → Draft queue lists inbound draft statuses.
- Evidence: `src/pages/DockIntakeReceiving.tsx:5-22`; `src/components/receiving/ReceivingStageRouter.tsx:159-170`; `src/hooks/useReceivingAutosave.ts:57-61`; `src/components/receiving/DraftQueueList.tsx:49-52`.

### Trace B — Stage 1 completion
`Stage1DockIntake.handleComplete` → flush autosave + persist exceptions → `shipments.update({inbound_status:'stage1_complete', ...})` → success toast + parent refresh.
- Evidence: `Stage1DockIntake.tsx:747-809`.

### Trace C — Stage 2 completion
`Stage2DetailedReceiving.handleComplete` → ensure unidentified account (if needed) + resolve location RPC → item create/update + exception/mismatch gates → `shipments.update({inbound_status:'closed', status:'received', received_at})` → billing preview snapshot + location assignment RPC + completion alert.
- Evidence: `Stage2DetailedReceiving.tsx:1262-1339`, `1351-1438`, `1573-1583`, `1635-1677`, `1697-1705`.

### Trace D — Receiving finalization (session variant)
`useReceivingSession.finishSession` → complete session row → update `shipments.status/received_at` → optional item/task generation → location RPC + alerts + coverage RPC.
- Evidence: `src/hooks/useReceivingSession.ts:189-209`, `240-488`, `496-563`.

Variant note: Stage2 UI and `useReceivingSession` finalize via different codepaths; both are active in repository (`[Partial]` harmonization risk). Evidence: `Stage2DetailedReceiving.tsx:1573-1583` vs `useReceivingSession.ts:204-209`.

## DR-4 STATUS MACHINE
DB-authoritative inbound status values and transitions:
- Allowed values constrained to `draft | stage1_complete | receiving | closed` (+ NULL). Evidence: `supabase/migrations/20260212130000_nvpc_receiving_state_machine.sql:22-31`.
- Trigger-enforced transitions:
  - `NULL -> draft`
  - `draft -> stage1_complete`
  - `stage1_complete -> receiving`
  - `stage1_complete -> draft`
  - `receiving -> closed`
  Evidence: `20260212130000_nvpc_receiving_state_machine.sql:49-59`.

Requested statuses `partial` and `cancelled` appear in UI theme but are not part of inbound_status DB state machine. Evidence: `src/components/receiving/StatusBar.tsx:4,55-72`; `[Risk]` taxonomy drift.
