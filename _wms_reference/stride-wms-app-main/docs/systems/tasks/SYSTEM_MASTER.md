## 1. FUNCTIONAL SCOPE

### Primary Purpose

Tasks is the work-order system of Stride WMS. It tracks every discrete unit of warehouse labor — inspection, assembly, repair, disposal, will-call pickup, item splitting, and custom work — from creation through completion, driving billing events and inventory status updates as side effects.

### Domains / Sub-domains

| Domain | Scope |
|--------|-------|
| Task lifecycle | Create → assign → start → complete / unable-to-complete / cancel |
| Task types & configuration | System-seeded and tenant-created task types with billing linkage |
| Billing integration | Task completion triggers billing events via `createTaskBillingEvents()` |
| Timer / labor tracking | Per-task timer intervals with offline support |
| Split workflow | Item quantity splitting with child item creation |
| Special flows | Will Call (pickup release), Disposal (item soft-delete) |
| Client portal | External clients can request tasks (pending_review) |

### Key Boundaries

- **Tasks does NOT own pricing.** Rates come from `charge_types` + `pricing_rules` via billing utilities. Tasks only references service codes on task types.
- **Tasks does NOT own invoicing.** It creates `billing_events`; the Billing system handles invoicing.
- **Tasks does NOT own item creation.** Split workflows create child items, but Items is the owner system.
- **Tasks does NOT own shipment lifecycle.** Tasks can be created from shipments, but Shipments owns its own status flow.

### Cross-System Touchpoints

| System | Interaction |
|--------|-------------|
| Billing | `task_completion` / `addon` billing events on complete; waive/override controls |
| Items / Inventory | `updateInventoryStatus()` sets assembly/inspection/repair status; `clearDamageAndQuarantine()` on repair; disposal sets `deleted_at` |
| Shipments | Tasks created from ShipmentDetail; split tasks block shipment processing |
| Receiving | `useReceivingSession.ts:430–484` auto-creates Inspection and/or Assembly tasks per item during receiving (gated by `shouldCreateInspections` / `shouldCreateAssembly` flags derived from tenant preferences or account settings) |
| Repair Quotes → Tasks | `useRepairQuotes.ts:396–418` auto-creates Repair task on quote approval via `createRepairTaskIfNeeded()`. Uses `resolveRepairTaskTypeId()` for 3-tier task type resolution |
| Client Portal | `ClientTaskCreate.tsx` and `ClientOutboundCreate.tsx` allow client-initiated tasks |
| Repair Quotes | `resolveRepairTaskType.ts` selects repair task types; repair quotes link to tasks via `repair_task_id` |
| Alerts | Task lifecycle events queue alerts (`queueTaskCreatedAlert`, `queueTaskAssignedAlert`, etc.) |
| Activity Logging | Per-item activity logged via `logItemActivity()` on task creation, start, and completion |

### Service Lines vs Custom Charges

Both stored in `task_custom_charges` but differentiated by `charge_description` JSON metadata:

- **Service Lines**: `charge_description` contains `{"is_service_line": true, "charge_type_id": "...", "input_mode": "qty|time|both", "minutes": N}`. Managed via `useTaskServiceLines.ts`.
- **Custom Charges**: `charge_description` is plain text or null. Managed via `useTaskCustomCharges.ts`. Can be created from `billing_charge_templates`.

### RLS / Permissions

- **Tables involved:** `tasks`, `task_types`, `subtasks`, `due_date_rules`, `billing_charge_templates`, `task_custom_charges`
- **Expected roles:** All authenticated users within the same tenant can read/write. No role-based restrictions beyond tenant isolation.
- **Policy file references:**
  - `tasks`: `supabase/migrations/20260121003907_b2b827fd-f75f-4f75-b837-585cfdde2bdf.sql` — uses `get_current_user_tenant_id()`

### Error Handling

| Flow | 400 (Validation) | 401 (Unauthenticated) | 403 (RLS Denial) | 500 (Server) |
|------|-------------------|----------------------|-------------------|---------------|
| Task creation | Toast: "Failed to create task" with `error.message` | Guards: `if (!profile?.tenant_id) return null` | RLS error surfaced as 500-style toast | Toast: "Failed to create task" |

## 5. SEQUENTIAL DATA FLOW

### Flow: Staff Task Creation

```
1. User opens TaskDialog (TaskDetail or Tasks page)
2. User fills title, type, priority, account, items, description
3. User clicks "Create Task"
4. TaskDialog calls useTasks.createTask(taskData, itemIds)
5. createTask() inserts into `tasks` table with status='pending'
6. If itemIds provided: inserts into `task_items` junction table
7. Calls updateInventoryStatus() → updates items.{assembly|inspection|repair}_status to 'pending'
8. Logs per-item activity via logItemActivity()
9. Queues queueTaskCreatedAlert()
10. Toast: "Task Created"
11. Calls fetchTasks() to refresh list
```

### Flow: Task Start (with Timer)

```
1. User clicks "Start Task" on TaskDetail
2. Handler calls startTaskDetailed(taskId, {pauseExisting})
3. If task has pending Split tasks → returns {ok: false, error_code: 'SPLIT_REQUIRED'}
4. Calls timerStartJob() → RPC rpc_timer_start_job (with offline fallback)
5. If timer fails → returns error, task not started
6. Updates tasks: status='in_progress', assigned_to=currentUser, started_at=now()
7. Clears pending_review metadata if present
8. Calls updateInventoryStatus() → items status to 'in_progress'
9. Queues queueTaskAssignedAlert()
10. UI updates to show in-progress state
```

### Flow: Normal Task Completion

```
1. User clicks "Complete Task" on TaskDetail
2. Handler calls completeTask(taskId) or completeTaskWithServices(taskId, values)
3. Computes estimated service time snapshot (non-blocking)
4. Calls computeTaskActualTimeSnapshot():
   a. Calls timerEndJob(reason='complete')
   b. Queries job_time_intervals for total labor minutes
   c. Returns ServiceTimeActualSnapshotV1
5. Updates tasks: status='completed', completed_at, completed_by, ended_at, duration_minutes
6. Merges time snapshots into tasks.metadata.service_time
7. Calls updateInventoryStatus() → items status to 'completed'
8. If Repair task: calls clearDamageAndQuarantine() → clears has_damage, removes quarantine flags
9. Calls createTaskBillingEvents():
   a. Checks waive_charges → skips if waived
   b. Checks for duplicate billing events → skips if exists
   c. Looks up primary_service_code (fallback: default_service_code)
   d. If no service code → non-billable, skip
   e. Determines billing unit (Task vs Item)
   f. Creates billing_events via createCharges()
10. Calls convertTaskCustomChargesToBillingEvents() for addon charges
11. Logs per-item activity
12. Queues queueTaskCompletedAlert() (or queueInspectionCompletedAlert for Inspection)
13. Toast: "Task Completed"
```

### Flow: Timer / Labor Time Lifecycle

```
Components:
  timerStartJob() — src/lib/time/timerClient.ts:37
  timerEndJob()   — src/lib/time/timerClient.ts:132
  flushOfflineTimers() — src/lib/time/timerClient.ts:183
  rpc_timer_start_job — PostgreSQL RPC
  rpc_timer_end_job   — PostgreSQL RPC
  offlineTimerQueue   — src/lib/time/offlineTimerQueue.ts (localStorage)

START (online):
1. startTaskDetailed() calls timerStartJob({..., pauseExisting})
2. timerStartJob() calls rpc_timer_start_job RPC
3. RPC inserts job_time_intervals row: started_at=now(), ended_at=NULL
4. If pauseExisting=true: RPC ends any active interval for same user (ended_reason='auto_pause')
5. Returns: {ok, started_interval_id, paused_interval_id, paused_job_type, paused_job_id}

START (offline):
1. timerStartJob() catches RPC failure, checks navigator.onLine
2. If offline: enqueues start event to offlineTimerQueue (localStorage)
3. Returns: {ok: true, offline: true, offline_started_at}

PAUSE (implicit only — no explicit pause button):
  - Pause occurs when starting a DIFFERENT job with pauseExisting=true
  - RPC ends current interval with ended_reason='auto_pause'
  - New interval starts for the new job
  - Paused job info returned: paused_job_id, paused_job_type

RESUME:
  - No explicit resume. User re-starts the paused task via startTaskDetailed()
  - This creates a NEW interval, not an extension of the paused one

STOP / END:
1. timerEndJob({..., reason}) called (default reason='pause')
2. On task completion: reason='complete'
3. RPC ends the active interval: ended_at=now(), ended_reason=reason
4. Returns: {ok, ended_interval_id}

STOP (offline):
1. timerEndJob() catches RPC failure
2. Enqueues end event to offlineTimerQueue

OFFLINE SYNC:
1. flushOfflineTimers() processes localStorage queue
2. For each queued event: calls appropriate RPC
3. Returns: {ok, inserted, remaining, hadActive}

ACTUAL TIME SNAPSHOT (on completion):
1. computeTaskActualTimeSnapshot() at useTasks.ts:1117
2. Calls timerEndJob(reason='complete') to close active interval
3. Queries all job_time_intervals for this task
4. Computes:
   - actual_labor_minutes: sum of (ended_at - started_at) across intervals
   - actual_cycle_minutes: first started_at to last ended_at
5. Returns ServiceTimeActualSnapshotV1

HOW TIME FEEDS BILLING QUANTITY:
  - For service lines with input_mode='time':
    billing quantity = minutes / 60 (hours)
    → useTasks.ts:1795–1796 (createServiceLineBillingEvents)
    → useTasks.ts:992–993 (computeTaskServiceTimeSnapshot)
  - For service lines with input_mode='qty':
    billing quantity = lineVal.qty (direct)
  - Primary billing (non-service-line): quantity determined by billing unit
    (Task=1, Item=per-item count) — time does NOT affect primary billing qty
```

2. Click **"Start Task"** — this starts the labor timer and assigns you
3. Perform the work
4. Add service lines if needed (service codes + qty/time)
5. Click **"Complete Task"**
   - If Will Call: enter **Pickup Name** in the dialog
   - If items required: ensure items are attached first
6. Billing events are created automatically

### How to Manage Task Types (Admin)

1. Navigate to **Settings** → **Task Types**
2. View all task types (System vs Custom badges)
3. To create: click **"Add Task Type"**, fill in name, kind, service category
4. To edit: click edit icon on existing type
5. To clone a system type: click clone icon
6. To deactivate: toggle the active switch

### How to Request a Task (Client Portal)

1. Navigate to **Client Portal** → **Create Task**
2. Select your account and items
3. Choose task type
4. If partial quantity: specify the quantity to keep vs split off
5. Submit — task enters **Pending Review** status
6. Warehouse staff will review and start the task
